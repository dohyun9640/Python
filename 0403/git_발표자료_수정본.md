# Git 입문: 버전 관리의 모든 것

---

## 1. 도입: 왜 버전 관리가 필요한가?

우리는 파일을 수정할 때 흔히 "최종.doc", "최종_진짜최종.doc", "최종_마지막_수정.doc" 같은 방식으로 관리하곤 합니다. 개인 작업이라면 어떻게든 버틸 수 있지만, 수천 개의 파일을 여러 명이 동시에 수정해야 하는 프로그래밍의 세계에서는 이런 방식이 불가능에 가깝습니다.

버전 관리 시스템(VCS, Version Control System)은 이 문제를 해결하기 위해 만들어졌습니다. 코드의 변경 이력을 자동으로 추적하고, 버그가 발생했을 때 과거의 특정 시점으로 즉시 되돌릴 수 있으며, 여러 개발자가 서로의 코드를 덮어쓸 걱정 없이 협업할 수 있도록 돕는 필수 인프라입니다.

그 중심에 Git이 있습니다. 현재 전 세계 전문 개발자의 약 93%가 Git을 사용하고 있을 만큼, Git은 선택이 아닌 필수 역량입니다.

> **핵심 한 줄**: Git은 코드의 "타임머신"이자 협업의 교통 정리 시스템이다.

---

## 2. Git의 핵심 철학: "스냅샷" 방식

Git을 이해하려면 먼저 Git이 데이터를 어떻게 저장하는지 알아야 합니다.

대부분의 구형 버전 관리 시스템은 파일의 차이점(Delta)만을 저장했습니다. "이 줄이 추가됐고, 저 줄이 삭제됐다"는 식의 기록입니다. 반면 Git은 전혀 다른 방식을 택했습니다.

Git은 커밋(commit = 기록)할 때마다 그 시점의 전체 파일 상태를 사진 찍듯이 통째로 저장합니다. 이를 "스냅샷(Snapshot)" 방식이라고 합니다. 물론 파일이 변경되지 않았다면 이전 버전의 링크만 저장해 효율성을 높입니다.

이 구조 덕분에 Git은 단순한 도구를 넘어 "콘텐츠 주소 지정이 가능한 파일 시스템"처럼 동작하며, 데이터의 무결성을 강력하게 보장합니다. 특정 커밋 하나만 있으면 그 시점의 프로젝트 상태를 완벽하게 복원할 수 있는 이유가 바로 여기에 있습니다.

> **핵심 한 줄**: Git은 파일의 차이가 아닌, 전체 상태의 스냅샷을 저장한다.

---

## 3. 반드시 이해해야 할 Git의 3가지 영역

Git을 처음 배울 때 가장 많이 혼란을 겪는 부분이 바로 '영역(Area)'의 개념입니다. Git은 파일을 다음 세 가지 상태로 관리합니다.

### Working Tree (작업 영역)
내 컴퓨터에서 실제로 코드를 수정하고 있는 디렉토리입니다. 파일을 만들고, 고치고, 지우는 모든 작업이 여기서 일어납니다.

### Staging Area (스테이징 영역)
다음 스냅샷(커밋)에 포함할 변경 사항들을 선택적으로 모아두는 '준비 영역'입니다. 이삿짐을 싸기 전, 박스에 넣을 물건을 미리 골라두는 과정과 같습니다. 이 영역이 있기 때문에 수정한 파일 중 일부만 선택적으로 커밋할 수 있습니다.

### Git Directory (저장소)
커밋이 완료되어 스냅샷이 로컬 데이터베이스에 영구적으로 안전하게 저장된 상태입니다. `.git` 폴더가 바로 이 역할을 합니다.

파일이 이동하는 방향은 항상 Working Tree → Staging Area → Git Directory 순입니다.

> **핵심 한 줄**: 수정(Modified) → 준비(Staged) → 기록(Committed), 이 세 단계가 Git의 기본 흐름이다.

## 4. 기본 명령어 흐름 익히기

실무에서 가장 자주 사용하는 기본 명령어 흐름입니다.

```bash
git init              # 현재 디렉토리를 Git 저장소로 초기화
git status            # 파일의 현재 상태 확인 (수정됨, 스테이징됨 등)
git add [파일명]       # 수정된 파일을 스테이징 영역으로 이동
git commit -m "메시지" # 스테이징 영역의 파일들을 스냅샷으로 기록
git log               # 지금까지의 커밋 히스토리 확인
```

여기서 가장 중요한 습관 하나는 명확한 커밋 메시지를 작성하는 것입니다. 나중에 히스토리를 되돌아볼 때, 혹은 팀원이 이력을 살펴볼 때, "수정함"보다 "로그인 실패 시 오류 메시지 표시 기능 추가"가 훨씬 유용합니다. 또한 너무 큰 작업을 한 번에 커밋하기보다, 논리적인 단위로 자주 커밋하는 것이 나중에 문제를 추적하기 훨씬 쉽습니다.

> **핵심 한 줄**: `add`로 고르고, `commit`으로 찍는다. 커밋 메시지는 미래의 나와 팀원에게 보내는 편지다.

## 5. 가지 치기와 합치기: Branch & Merge

Git의 진짜 강력함은 브랜치(Branch)에서 나옵니다.

브랜치는 메인 코드에 영향을 주지 않고 새로운 기능을 독립적으로 개발할 수 있는 '가벼운 포인터'입니다. 예를 들어, 서비스가 운영 중인 상황에서 새 기능을 개발하거나 버그를 수정할 때 브랜치를 만들어 작업하면, 메인 코드는 안정적으로 유지되고 작업이 완료된 후에 합치면 됩니다.

```bash
git branch [브랜치명]     # 새 브랜치 생성
git checkout -b [브랜치명] # 브랜치 생성 + 즉시 이동 (자주 쓰는 단축 명령어)
git merge [브랜치명]       # 현재 브랜치에 다른 브랜치를 합치기
```

브랜치를 합치는 방법에는 두 가지가 있습니다.

**Merge**: 두 갈래로 나뉜 작업 내용을 하나로 합칩니다. 변경 이력이 그대로 보존되어 "언제 어떤 브랜치에서 어떤 작업을 했는지" 흔적이 남습니다.

**Rebase**: 브랜치가 시작된 지점을 최신 상태로 옮겨 선형적인(일직선의) 히스토리를 만드는 방식입니다. 기록이 깔끔해지는 장점이 있지만, 이미 팀원과 공유된 브랜치에서 사용하면 히스토리가 뒤바뀌어 심각한 혼란을 초래할 수 있습니다. 아직 공유하지 않은 내 로컬 브랜치에서만 사용하는 것을 권장합니다.

> **핵심 한 줄**: 브랜치로 실험하고, Merge로 합친다. 공유된 브랜치에서의 Rebase는 금물이다.

## 6. Git vs GitHub: 도구와 플랫폼의 차이

많은 입문자가 Git과 GitHub을 혼동합니다. 이 둘은 엄연히 다른 것입니다.

**Git**은 내 컴퓨터에서 돌아가는 버전 관리 도구(엔진)입니다. 인터넷이 없어도 작동합니다.

**GitHub**은 Git 저장소를 클라우드에 호스팅하고 전 세계 개발자와 협업할 수 있게 해주는 플랫폼입니다. Git이 엔진이라면 GitHub은 그 엔진을 얹은 자동차라고 볼 수 있습니다. GitHub 외에도 GitLab, Bitbucket 등 유사한 플랫폼이 존재합니다.

```bash
git push   # 내 로컬 커밋을 원격 저장소(GitHub 등)에 업로드
git pull   # 원격 저장소의 변경 사항을 가져와서 자동으로 합치기
git fetch  # 원격 저장소의 변경 사항을 가져오기만 하고, 합치지는 않음
```

`pull`과 `fetch`의 차이를 짚어두면 좋습니다. `pull`은 변경 사항을 가져오는 동시에 내 로컬 브랜치에 자동으로 합쳐(merge)버리지만, `fetch`는 가져오기만 하고 합치는 것은 내가 직접 결정합니다. 어떤 변경 사항인지 먼저 확인하고 싶을 때 `fetch`가 유용합니다.

GitHub의 핵심 협업 메커니즘은 Pull Request(PR)입니다. 내가 작업한 브랜치를 메인 브랜치에 합쳐달라고 팀원들에게 공식적으로 요청하는 과정으로, 코드 리뷰와 토론이 이 PR을 중심으로 이루어집니다.

> **핵심 한 줄**: Git은 도구, GitHub은 플랫폼. Pull Request는 팀 협업의 심장이다.

## 7. 보안 및 관리 베스트 프랙티스

코드를 잘 관리하는 것과 보안은 떼려야 뗄 수 없습니다. 특히 다음 두 가지는 입문 단계부터 반드시 습관으로 들여야 합니다.

**`.gitignore` 파일 활용**: API 키, 비밀번호, DB 접속 정보 같은 민감한 정보가 코드에 포함된 채 커밋되면 심각한 보안 사고로 이어질 수 있습니다. `.gitignore` 파일에 추적하지 않을 파일이나 폴더를 명시해두면 Git이 해당 파일을 아예 무시합니다. 빌드 결과물, 의존성 폴더(`node_modules` 등), 환경 설정 파일(`.env`) 등도 이 파일에 등록하는 것이 일반적입니다.

**작은 단위의 커밋**: 큰 작업을 한 번에 커밋하는 것보다, 논리적으로 의미 있는 작은 단위로 자주 커밋하는 것이 훨씬 좋습니다. 나중에 버그가 발생했을 때 어느 커밋에서 문제가 생겼는지 빠르게 추적할 수 있고, 특정 변경 사항만 되돌리기도 쉬워집니다.

> **핵심 한 줄**: `.gitignore`로 비밀을 지키고, 작은 커밋으로 이력을 명확하게 남긴다.

## 8. 마무리: Git의 미래와 학습 방향

Git은 현재도 활발히 발전하고 있습니다. 더 강력한 보안을 위해 기존 SHA-1 해시 알고리즘을 SHA-256으로 전환하는 작업이 진행 중이며, 수백만 개의 파일을 다루는 대규모 모노레포(monorepo) 환경의 성능을 획기적으로 개선하기 위한 Reftable 등의 기술도 도입되고 있습니다.

Git을 처음 배울 때는 CLI(터미널) 명령어가 낯설고 두렵게 느껴질 수 있습니다. GUI 도구(GitHub Desktop, Sourcetree 등)를 함께 활용하는 것도 좋은 방법이지만, 결국 기본 원리와 명령어를 이해하는 것이 중요합니다. GUI는 내부에서 같은 명령어를 실행하기 때문입니다.

추천하는 학습 순서는 다음과 같습니다.

1. 로컬에서 `init`, `add`, `commit`, `log` 반복 실습
2. 브랜치 만들고 합치는 연습 (`branch`, `checkout`, `merge`)
3. GitHub에 저장소 만들고 `push`, `pull` 경험
4. 팀 프로젝트에서 Pull Request 흐름 익히기

기본 원리를 이해하고 꾸준히 실습한다면, Git은 여러분의 개발 여정에서 가장 강력한 동반자가 될 것입니다.

> **핵심 한 줄**: 두려워하지 말고 일단 써보자. 실수해도 Git이 되돌려준다.
